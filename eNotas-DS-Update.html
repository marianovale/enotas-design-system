Contexto

O problema que resolvemos

Antes do DS, cada view era uma ilha. Estilos duplicados, interações inconsistentes e zero rastreabilidade de design.

Antes

Sem design system · Razor Views sem padrão

  • CSS duplicado em cada .cshtml — sem tokens compartilhados
  • Botões e badges reescritos à mão em cada view
  • Nenhuma ligação entre Figma e o código implementado
  • Mudança de cor exigia buscar arquivo por arquivo
  • Sem cobertura de acessibilidade ou estados de foco
  • Onboarding de devs dependia de alguém explicar os padrões
  • Deploy manual, sem pipeline de versionamento

Agora

eNotas DS · Web Components + tokens centralizados

  • Tokens CSS centralizados com prefixo --en-*
  • <en-button> roda em qualquer .cshtml via script tag
  • Code Connect: Dev Mode do Figma entrega o código real
  • Mudança de cor em 1 arquivo → propaga em toda a app
  • Shadow DOM garante encapsulamento e sem colisão de estilos
  • Storybook documenta estados, props e exemplos de uso
  • CI/CD no GitHub Actions — build e preview automáticos

Como o Claude Code foi parte do time

O que isso muda para PDs

A barreira entre "design" e "implementação" não existe mais da mesma forma

O Claude Code não substitui engenheiros — ele reduz a dependência de ter um engenheiro disponível para cada decisão de infraestrutura. Um PD com contexto do produto pode agora fazer perguntas técnicas, receber respostas compreensíveis, e executar mudanças que antes ficavam na fila de backlog.

O que o Claude não fez
  • Decidiu quais componentes o eNotas precisava — isso veio do entendimento do produto
  • Definiu a estratégia de migração para o Cosmos — isso exigiu entender o roadmap da Hotmart
  • Escolheu alinhar com o Cosmos desde o início — essa foi uma decisão de negócio, não técnica

A IA executa. O PD decide o que, por que e para quem.

2

Resultado: zero tempo de interpretação de design → menos erros, mais velocidade. O código que o dev copia no Figma é exatamente o código que roda na view.

Fundação

Tokens de design

Uma única fonte de verdade para cores, espaçamentos, tipografia e sombras — gerada via Style Dictionary a partir do Figma.

Arquitetura de dois níveis

Uma mudança de cor primária — de teal para qualquer outro tom — acontece em um único lugar e se propaga automaticamente para todos os componentes.

Nível Exemplo Função
Global (primitivo) --en-color-teal-500: #22baa0 Valor bruto, sem semântica — base da escala de cores
Semântico (alias) --en-action-primary-background: var(--en-color-teal-500) Intenção de uso — o que os componentes consomem

Roadmap

Caminho para o Cosmos

A estratégia de migração já está embutida na arquitetura do DS.

Hoje

.NET / Razor Views

  • Web Components via script tag
  • Shadow DOM — zero colisão de CSS
  • jQuery/legacy compatível
  • Code Connect + Figma MCP ativos
  • CI/CD + Storybook online
Curto prazo

Alinhamento + React

  • Wrappers React gerados pelo Stencil
  • Publicar no npm (hotmart-org)
  • Alinhar tokens restantes com Cosmos
  • Testes de acessibilidade (a11y)
Médio prazo

npm + Adoção

  • Pacote público no npm da Hotmart
  • Integração com Chromatic para visual regression
  • Documentação expandida no Storybook
  • Onboarding de outros squads
Futuro

Merge Cosmos DS

  • Substituição por componentes Cosmos
  • Find-and-replace nas views .cshtml
  • Tokens semânticos remapeados
  • Contribuição upstream para Cosmos
Nota arquitetural: A API dos componentes já está alinhada com o Cosmos DS desde o início — props como variant, size, intent e disabled seguem os mesmos nomes. Migrar será um find-and-replace, não uma reescrita.
Como a migração acontecerá na prática
Etapa 1 — Hoje (concluída)
.NET / Razor via <script> tag
O DS roda no .NET via script tag. 30 componentes disponíveis. O HTML das views permanece intacto.
<!-- _Layout.cshtml — uma vez no layout base -->
<link rel="stylesheet" href="/dist/tokens.css">
<script type="module"
  src="/dist/enotas-ds/enotas-ds.esm.js"></script>

<!-- Em qualquer .cshtml -->
<en-button variant="primary">Emitir Nota</en-button>
Etapa 2 — Alinhamento de tokens
Arquivo de compatibilidade
Quando o time Cosmos definir o mapeamento semântico, basta um arquivo de alias. O HTML das views não muda nada.
/* arquivo de compatibilidade */
--en-action-primary-background:
  var(--cosmos-action-primary);

--en-feedback-positive:
  var(--cosmos-feedback-success);
Etapa 3 — Migração gradual para React
Views .cshtml → React
Views migradas para React simplesmente trocam a tag HTML pelo wrapper React. A lógica de negócio continua idêntica.
<!-- Razor (continua funcionando) -->
<en-button variant="primary">
  Emitir
</en-button>

// React/Cosmos (novo — mesmo componente)
<EnButton variant="primary">
  Emitir
</EnButton>
Etapa 4 — Unificação (opcional)
Merge com o Cosmos DS
Se o Cosmos absorver os componentes, a API de props já está alinhada. Toda a migração se resume a um find-and-replace de tags.
// O que NÃO precisará ser refeito:

// ✓ Tokens (só o mapeamento semântico muda)
// ✓ API de props (variant, size, intent)
// ✓ Stories do Storybook
// ✓ Code Connect no Figma
Antes × Agora

Resumo executivo

O eNotas DS não é um projeto paralelo ao produto — é a infraestrutura que permite que qualquer evolução de interface aconteça mais rápido, com mais consistência e com menos risco de regressão visual.

Dimensão Situação antes Situação agora
Consistência visual Componentes ad-hoc por view — cada tela reinventava botões, badges e inputs 30 componentes unificados, baseados nos tokens do Figma
Velocidade de desenvolvimento Dev interpretava o design manualmente e reescrevia HTML a cada sprint Code Connect gera o código do componente certo direto no Dev Mode do Figma
Compatibilidade com .NET Não havia camada de componentes — CSS solto em cada .cshtml Web Components nativos — funcionam via <script> tag, zero configuração
Dívida técnica Qualquer componente React criaria uma bifurcação do stack — reescrita inevitável Stencil gera wrappers React automaticamente — sem bifurcação
Merge com Cosmos Dependia de reescrever o frontend completo para React antes de convergir Troca de mapeamento de tokens + find-and-replace nas tags das views
Documentação Inexistente — onboarding dependia de alguém explicar os padrões Storybook com Controls, acessibilidade e Code Connect publicado
Processo

Governança e evolução

O DS é vivo. Qualquer dev do time pode evoluir componentes ou tokens seguindo o processo abaixo.

Para adicionar um novo componente

1

Inspecionar no Figma Dev Mode

Verificar variants, props e estados disponíveis antes de implementar

2

Criar en-[nome].tsx + .css

Lógica Stencil com props tipadas + estilos usando exclusivamente tokens --en-*

3

Criar en-[nome].stories.ts

Documentação interativa no Storybook com Controls e casos de uso

4

Criar en-[nome].figma.tsx

Mapeamento Code Connect — conecta o componente Figma ao código real

5

Build e publicação

pnpm build:components && pnpm figma:connect

6

Atualizar STATUS.md

Registrar o novo componente, sua tag, casos de uso e data de entrega

O Stencil gera documentação de props automaticamente — sem esforço manual de manutenção de docs.

Para atualizar tokens do Figma

1

Exportar variáveis do Figma

Usar o plugin de exportação ou o Figma MCP para extrair as variáveis atualizadas

2

Atualizar packages/tokens/src/

Editar global/ para valores primitivos ou semantic/ para aliases de intenção

3

Rodar o build de tokens

pnpm build:tokens — Style Dictionary regenera o tokens.css

4

Propagação automática

Os componentes já refletem a mudança — CSS custom properties se propagam automaticamente via Shadow DOM

Integração com jQuery — zero adaptação

$('en-button').on('enClick', function() {
  // lógica jQuery existente intacta
});

Os eventos dos Web Components funcionam com jQuery como qualquer evento DOM nativo.
